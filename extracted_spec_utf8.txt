Graviton Game Room System Feature Specification ΓÇö Build Document Version 1.0 | For AI Agent Implementation 1. Overview This document specifies the complete Game Room system for the Graviton Social Gaming Portal. It covers the data model, room lifecycle, host flow, guest flow, real-time behaviour, coin escrow logic, and UI requirements. Build exactly what is described here ΓÇö no feature should be assumed or added beyond this spec. Core principle A game room is a temporary session, not a permanent space. It is created by a host, exists only while the host is present, and dissolves completely when the session ends. No host = no room. 1.1 What the game room system does Allows a user (the host) to create a private, temporary room for playing a game with friends Supports coin stakes ΓÇö players can put digital currency on the line before a match Uses real-time presence so all players see each other&apos;s status live in the lobby Integrates with the admin portal, which configures the available modes and stakes options Dissolves automatically after the game ends or the host cancels 1.2 What it does NOT do There are no permanent rooms or persistent group spaces There is no Public Rooms feature ΓÇö do not build this A user cannot host more than one active room simultaneously Guests cannot transfer host privileges ΓÇö the room owner is always the creator 2. Data model All game room data lives in Supabase. The following tables and columns are required. 2.1 game_rooms table Column Type Notes id uuid PK Auto-generated name text Room display name set by host host_id uuid FK References auth.users. Deleting the host dissolves the room. game_id uuid FK References games table. Must always be set ΓÇö never null after creation. mode enum Values: friend_room | online_ranked | practice status enum Values: forming | open | live | dissolved stake_amount integer Coin amount per player. 0 = free play. join_code text 6-character uppercase alphanumeric. Unique. Auto-generated. max_players integer Default 4. Configurable by admin per game. created_at timestamptz Auto-set on insert. dissolved_at timestamptz Null until dissolved. Set on cancel or game end. 2.2 room_players table One row per player per room. Tracks lobby presence and readiness. Column Type Notes id uuid PK Auto-generated room_id uuid FK References game_rooms.id user_id uuid FK References auth.users is_ready boolean Default false. Reset to false if host changes game or stakes. joined_at timestamptz When the player entered the lobby. left_at timestamptz Null until player leaves or is kicked. 2.3 room_invites table Tracks pending invitations. Expires when the room dissolves. Column Type Notes id uuid PK Auto-generated room_id uuid FK References game_rooms.id inviter_id uuid FK The host&apos;s user ID invitee_id uuid FK The invited friend&apos;s user ID status enum Values: pending | accepted | declined | expired created_at timestamptz Auto-set on insert 2.4 coin_transactions table Append-only ledger for all coin movements. Never update or delete rows. Column Type Notes id uuid PK Auto-generated user_id uuid FK The user whose balance is affected room_id uuid FK The room this transaction belongs to type enum Values: escrow | release_win | release_cancel amount integer Positive = credit, Negative = debit created_at timestamptz Auto-set on insert 3. Room lifecycle Every room passes through exactly four states. The status column in game_rooms tracks this. State Description What triggers the next state forming Host is going through the creation wizard. Room exists in DB but is not visible to other users. Host completes all wizard steps and confirms. open Lobby is live. Guests can join via invite. Host can still change game or stakes (resets all ready statuses). Host clicks Start and all players are marked is_ready = true. live Game is in progress. Coins are escrowed. Room is locked ΓÇö no new players can join. Game ends (natural completion) or host force-closes. dissolved Room is over. No further actions possible. Record is retained for audit but treated as inactive. Reached from open (cancel) or live (game end / host disconnect). 3.1 Dissolve triggers The following events dissolve a room. Each must be handled explicitly in code. Trigger Coin behaviour What guests see Host cancels from lobby (status = open) No escrow has occurred. No coin movement needed. Real-time toast: &quot;Host ended the room.&quot; Redirected to /dashboard/gameroom. Game completes normally (status = live) Escrow released. Winner receives full pot. Losers&apos; coins are not returned. Result screen shown with final scores and wallet update. Host disconnects during live game 60-second grace period. If host reconnects within 60s, room continues. After 60s: escrow is reversed (release_cancel), all players refunded, room dissolved. Countdown banner: &quot;Host disconnected ΓÇö room closing in Xs.&quot; Redirect after countdown. Host cancels from live game (force quit) Escrow reversed immediately. All players refunded via release_cancel transactions. Toast: &quot;Host ended the game early. Coins returned.&quot; Redirect to /dashboard/gameroom. 4. Host flow ΓÇö create room wizard The host creates a room through a 4-step wizard. Each step must be completed before advancing. The wizard creates the room record in the DB at step 1 with status = forming. Status updates to open when the host enters the lobby. UX rule Each screen has exactly one job. Do not combine steps. Do not show the full form at once. Step 1 ΓÇö Choose room mode Route /dashboard/gameroom/create/mode Purpose Host selects which type of room to create. This drives all downstream options. Options friend_room: Private, flexible stakes, no leaderboard impact online_ranked: Public matchmaking, affects global leaderboard practice: Solo/bot opponents, no coins, no rankings Source of options Mode options are configured in the admin portal. Fetch from admin config ΓÇö do not hardcode. Validation A mode must be selected before Next is enabled. DB action Insert row into game_rooms with host_id, mode, status = forming. Step 2 ΓÇö Pick a game Route /dashboard/gameroom/create/game Purpose Host selects which game the room will play. Source Fetch from games table where is_published = true. Render as a grid of cards showing game name, category tag, and thumbnail. Validation A game must be selected before Next is enabled. game_id must never be null when room status leaves forming. DB action Update game_rooms.game_id for this room. Step 3 ΓÇö Set stakes Route /dashboard/gameroom/create/stakes Purpose Host sets the coin amount each player must commit. Options Fetch available stake amounts from admin config for the selected mode. Always include Free play (0 coins) as the first option. Validation A stake option must be selected before Next is enabled. Show the host&apos;s current coin balance. If stake amount exceeds balance, show a warning but still allow selection (guest may also be blocked at join time). DB action Update game_rooms.stake_amount for this room. Note For practice mode, skip this step entirely ΓÇö stakes are always 0. Step 4 ΓÇö Invite friends Route /dashboard/gameroom/create/invite Purpose Host invites friends into the room before entering the lobby. Friend list rules Show only friends where presence status = online. Friends in another live game show as Busy and their invite button is disabled. Friends who are offline are not shown. Invite action Tapping Invite on a friend: inserts a row into room_invites with status = pending. Sends a real-time notification to the invitee via Supabase real-time channel. Button changes to Invited with a checkmark. Host can un-invite before entering lobby. Advance without inviting Host can skip inviting and open the lobby immediately. The lobby stays open for latecomers. DB action Insert rows into room_invites for each invited friend. 4.1 Lobby ΓÇö host view Route /dashboard/gameroom/[roomId]/lobby Trigger Host is redirected here after Step 4. Status updated to open. The lobby shows the room&apos;s current state in real-time. It must update without page refresh using Supabase real-time listeners on the room_players table. Room summary strip Always visible at top of lobby Shows: game name, stake amount, room mode Game and stakes have inline &apos;change&apos; links (see Section 4.2) Player list Shows all players currently in room_players for this room Each row: avatar, display name, ready badge (Ready or Waiting) Host is always shown first with a &apos;(host)&apos; label Updates in real-time as guests join, leave, or toggle ready Start button Disabled until ALL players in room_players have is_ready = true When enabled: label is &apos;Start game&apos;. When disabled: label is &apos;Waiting for players&apos; On click: triggers coin escrow (see Section 6), updates room status to live, launches game for all players Cancel room button Always visible in lobby On click: show inline confirmation modal (not a page navigation) Confirmation text: &apos;End this room? All players will be removed and any pending invites will expire.&apos; Confirm action: set room status to dissolved, remove all room_players rows, expire all pending invites, redirect host to /dashboard/gameroom 4.2 Changing game or stakes from lobby Important If the host changes the game or stakes after players have already marked ready, all players must be reset to is_ready = false automatically. Change game: show an inline panel with the game list. On select, update game_rooms.game_id, reset all room_players.is_ready to false. Broadcast a real-time event to all connected clients. All guest screens show a banner: &apos;The host changed the game ΓÇö please re-confirm your ready status.&apos; Change stakes: same behaviour as change game but for stake_amount. The change panels are inline (within the lobby page). Do not navigate away from the lobby to change settings. 5. Guest flow 5.1 Receiving an invite When a host invites a friend, the friend receives a real-time notification. This notification must appear without a page refresh, wherever the guest currently is in the app. Notification content Room name, game name, stake amount, mode, host display name Notification actions Join lobby (primary action) | Decline (secondary action) On Join lobby Insert a row into room_players for the guest. Update room_invites.status to accepted. Redirect guest to /dashboard/gameroom/[roomId]/lobby. On Decline Update room_invites.status to declined. Dismiss notification. Host is not actively notified of decline ΓÇö they see the player count simply not increase. Invite expiry If room status changes to dissolved while invite is pending, the notification is dismissed and invite status set to expired. 5.2 Guest lobby view The guest lobby is the same route as the host lobby (/dashboard/gameroom/[roomId]/lobby) but renders differently based on whether the current user is the host. Difference from host view No &apos;Start game&apos; button. No &apos;change game/stakes&apos; links. Shows &apos;Mark as ready&apos; button instead. Stakes notice Display a banner: &apos;Marking ready means you agree to [X] coins being escrowed when the host starts the game.&apos; Only show if stake_amount &gt; 0. Mark as ready On click: set room_players.is_ready = true for the guest. Button changes to &apos;Unready&apos;. Guest can toggle back to not ready before host starts. Ready reset If host changes game or stakes, guest&apos;s is_ready is reset to false server-side. Guest sees a banner explaining why. They must re-confirm. Leave room Guest can leave the lobby at any time. Sets room_players.left_at timestamp. Host sees the slot free. Room continues ΓÇö it does not dissolve because a guest left. Real-time All player ready statuses update live without refresh. 6. Coin escrow logic Rule Coins are escrowed at the moment the host clicks Start, not when players mark ready. If stake_amount = 0, skip all escrow logic entirely. 6.1 Escrow on Start Verify all players in room_players have is_ready = true. If any are false, reject the Start action and show an error. For each player in room_players, verify their coin balance &gt;= stake_amount. If any player has insufficient funds, block Start and show which player(s) cannot afford the stake. In a single Supabase transaction: deduct stake_amount from each player&apos;s coin balance. Insert a coin_transactions row for each player with type = escrow and amount = -stake_amount. Update game_rooms.status to live. Broadcast real-time event to all clients to launch the game. 6.2 Escrow release ΓÇö winner When the game ends, the game client reports the result to the server (winner user_id). Calculate total pot: stake_amount x number of players in room_players. Credit the full pot to the winner&apos;s coin balance. Insert a coin_transactions row for the winner with type = release_win and amount = total pot. Update game_rooms.status to dissolved. Redirect all clients to the result screen. 6.3 Escrow release ΓÇö cancel Triggered by: host cancels from live game, or host disconnect grace period expires. For each player in room_players: credit stake_amount back to their coin balance. Insert a coin_transactions row for each player with type = release_cancel and amount = +stake_amount. Update game_rooms.status to dissolved. Broadcast real-time dissolve event to all clients. 7. The following events must be broadcast and handled. Event Trigger Client action player_joined Guest enters lobby (room_players insert) Update player list, show new avatar and Waiting badge player_ready Guest toggles is_ready (room_players update) Update player&apos;s badge from Waiting to Ready or vice versa. Enable Start if all ready. player_left Guest leaves lobby (room_players update left_at) Remove player from list. Disable Start if it was enabled. room_settings_changed Host updates game_id or stake_amount Show settings-changed banner on all guest screens. Reset all ready badges to Waiting. game_starting Host clicks Start, escrow complete All clients transition to game view simultaneously. room_dissolved Room status set to dissolved for any reason All clients redirected to /dashboard/gameroom. Show appropriate toast message. invite_received New row in room_invites for this user Show real-time invite notification anywhere in app. 9. Row Level Security (RLS) policies All tables must have RLS enabled. The following policies define who can read and write each table. game_rooms SELECT: User can read a room if they are the host_id, OR they have a row in room_players for this room, OR they have a pending/accepted invite in room_invites for this room. INSERT: Authenticated users only. User can only set themselves as host_id. UPDATE: Only the host (host_id = auth.uid()) can update the room. DELETE: Not allowed. Use status = dissolved instead. room_players SELECT: User can read rows where room_id matches a room they have access to (same rule as game_rooms SELECT). INSERT: User can only insert a row for themselves (user_id = auth.uid()). Room must have status = open. UPDATE: User can only update their own row (is_ready toggle). Host can update any row in their room. room_invites SELECT: User can read rows where invitee_id = auth.uid() OR inviter_id = auth.uid(). INSERT: Only the room host can insert invites for their room. UPDATE: Only the invitee can update status (to accepted or declined). coin_transactions SELECT: User can only read rows where user_id = auth.uid(). INSERT: Only server-side functions (not client). All escrow logic runs in Supabase Edge Functions or database functions, never from the client. UPDATE/DELETE: Not allowed. This is an append-only ledger. 10. Admin portal integration The admin portal configures options that the game room system consumes. Do not hardcode any of the following values ΓÇö always fetch from the admin config. Available room modes (friend_room, online_ranked, practice) and whether each is enabled Stake amount options per mode (e.g. friend_room: [0, 50, 200, 500]) Maximum stake amount per mode Whether a mode affects the leaderboard (leaderboard_impact: boolean) Published games available in the game picker (is_published = true in games table) Max players per game 11. Edge cases and error states User tries to join a full room If room_players count &gt;= game&apos;s max_players, reject the join and show error: &apos;This room is full.&apos; User tries to join a live room If room status = live, reject the join and show error: &apos;This game has already started.&apos; Player has insufficient coins at Start Block the Start action. Show a message identifying which player(s) cannot cover the stake. Host must either lower the stakes or wait for the player to top up. Host refreshes or closes the tab during lobby Room stays open. Supabase presence channel detects host absence. After 60 seconds with no reconnection: dissolve the room, refund any escrowed coins, notify all guests. Guest refreshes during lobby Guest is reconnected to the same lobby. Their is_ready status is preserved from the DB. Duplicate join attempts If a user already has a row in room_players for this room, do not insert a new row. Re-subscribe them to the real-time channel and redirect to the lobby. 12.
